## 适用范围

**所有飞书多维表格操作**——抖音运营、设计素材库、视频管理、ERP数据、项目管理、客户管理……无论什么业务场景，建表/改表时必须应用本规范的全部能力。

## 触发
当用户说"创建飞书表格"、"设计表格"、"表格美化"、"多维表格怎么做"、"飞书表格怎么设计得好看"时。当需要输出数据到飞书时。当需要优化已有表格时。

## 目标
让每一张飞书多维表格从诞生起就好看、好用，而不是建完再改。

---

## 核心原则

### 1. 数据库思维 > Excel 思维
- Excel 思路：一列一个属性 → 表越来越宽
- 多维表格思路：把信息拆成独立的**实体表**，用**关联字段**连接
- 一个实体一张表（商品表、订单表、问题表），不要全堆一起

### 2. 字段类型决定第一印象
别人看表第一眼不是看数据，是看**字段怎么显示的**——裸数字 vs 货币符号 vs 进度条 vs 星星，体验天差地别。

### 3. 视图即角色，视图即场景
一张表至少 2 个视图：**管理视图**（摘要）+ **数据视图**（全量）。不同人看不同视图，不用复制表。

## 全部字段类型速查表（28 种）

### 基础字段

| type | ui_type | 中文名 | 何时用 | 创建示例 |
|------|---------|--------|--------|---------|
| 1 | Text | 文本 | 名称、备注等自由文字 | `{"field_name":"备注","type":1}` |
| 2 | Number | 数字 | 可计算数值 | `{"field_name":"数量","type":2}` |
| 2 | Currency | 货币 | 金额（自动¥+千分位） | `{"field_name":"GMV","type":2,"ui_type":"Currency","property":{"decimal_places":2}}` |
| 2 | Progress | 进度 | 百分比进度条 | ⚠️ API 不支持直接创建，先建 Number 再到 UI 改类型 |
| 2 | Rating | 评分 | 星级评分 | `{"field_name":"评分","type":2,"ui_type":"Rating","property":{"rating":{"max":5,"icon":"star"}}}` |
| 5 | DateTime | 日期 | 日期时间 | `{"field_name":"日期","type":5,"property":{"format":"yyyy-mm-dd"}}` |
| 7 | Checkbox | 复选框 | 是/否二元选择 | `{"field_name":"已确认","type":7}` |

### 选择类

| type | ui_type | 中文名 | 何时用 | 创建示例 |
|------|---------|--------|--------|---------|
| 3 | SingleSelect | 单选 | 状态、分类、优先级 | `{"field_name":"状态","type":3,"property":{"options":[{"name":"待处理","color":1}]}}` |
| 4 | MultiSelect | 多选 | 多标签 | `{"field_name":"标签","type":4,"property":{"options":[{"name":"ins风","color":1}]}}` |

颜色码: 0=灰, 1=红, 2=黄, 3=绿, 4=蓝, 5=紫

### 联系类

| type | ui_type | 中文名 | 何时用 | 创建示例 |
|------|---------|--------|--------|---------|
| 11 | User | 人员 | 负责人、协作者（支持@mention） | `{"field_name":"负责人","type":11}` |
| 12 | Email | 邮箱 | 邮件地址 | `{"field_name":"邮箱","type":12}` |
| 13 | Phone | 电话 | 手机号 | `{"field_name":"电话","type":13}` |

⚠️ 人员字段写入格式: `[{"id":"ou_xxx"}]`（需要飞书用户的 open_id，不能直接写文字姓名）

### 链接与引用类

| type | ui_type | 中文名 | 何时用 | 创建示例 |
|------|---------|--------|--------|---------|
| 15 | Url | 超链接 | 网址、飞书文档链接 | `{"field_name":"来源链接","type":15}` |
| 17 | Attachment | 附件 | 图片、文件 | `{"field_name":"图片","type":17}` |
| 18 | SingleLink | 单向关联 | A表引用B表 | `{"field_name":"关联指标","type":18,"property":{"table_id":"tblXXX"}}` |
| 19 | Lookup | 查找引用 | 从关联表自动拉字段 | 复杂配置，需指定关联字段+源字段+条件 |
| 21 | DuplexLink | 双向关联 | AB表互相引用 | 同 type 18 格式 |

⚠️ 附件字段写入需两步: 1) 上传到飞书Drive获取 file_token 2) 写入`[{"file_token":"xxx"}]`
⚠️ 公式(type 20)和查找引用(type 19)是**只读字段**，不能通过 API 写入数据

### 智能类

| type | ui_type | 中文名 | 何时用 | 创建示例 |
|------|---------|--------|--------|---------|
| 20 | Formula | 公式 | 自动计算、条件判断 | `{"field_name":"评级","type":20,"property":{"formula_expression":"IF([退款率]>20,\"🔴\",\"🟢\")"}}` |
| 25 | AutoNumber | 自动编号 | 流水号自增 | `{"field_name":"编号","type":25}` |
| 1002 | — | AI字段 | AI分类/提取/翻译/总结 | 需在飞书 UI 中配置 |

### 其他

| type | ui_type | 中文名 | 用途 |
|------|---------|--------|------|
| 22 | Location | 位置 | 地图位置 |
| 23 | GroupChat | 群聊 | 关联飞书群 |
| 1004 | CreatedUser | 创建人 | 自动记录 |
| 1005 | ModifiedUser | 修改人 | 自动记录 |

### 公式字段注意事项

- 引用字段用 `[字段名]`，**必须精确匹配字段名**（含括号/符号）
- 如字段名为 `退款率-支付口径（%）`，公式写 `[退款率-支付口径（%）]`
- 公式不可跨表引用，但可以引查找引用字段的值
- 常用函数: `IF()`, `IFS()`, `AND()`, `OR()`, `ROUND()`, `SUM()`, `AVERAGE()`
- 公式表达式在**创建时必须正确**，创建后可以但不好改

### Progress 字段说明

API 创建 Progress 字段可能报错 `WrongRequestBody (1254001)`。此时的做法:
1. 创建为普通 Number 字段: `{"field_name":"执行进度","type":2}`
2. 写入 0-1 范围的数值（0=0%, 0.5=50%, 1=100%）
3. 在飞书 UI 中将字段类型手动改为"进度"

### 字段类型决策流程

```
这个字段是什么性质?
├─ 金额 → Currency (type 2, ui_type Currency)
├─ 百分比/比率 → Progress (type 2, 先建Number再UI改)
├─ 评分/等级 → Rating (type 2, ui_type Rating)
├─ 状态/分类(单选) → SingleSelect (type 3) + 颜色
├─ 多标签 → MultiSelect (type 4)
├─ 是/否 → Checkbox (type 7)
├─ 日期 → DateTime (type 5)
├─ 人 → User (type 11)
├─ 网址 → Url (type 15)
├─ 图片/文件 → Attachment (type 17)
├─ 自动计算 → Formula (type 20)
├─ 引用其他表 → SingleLink (type 18) + Lookup (type 19)
└─ 自由文字 → Text (type 1)
```

## 跨表联动架构

### 何时拆表

一张表超过 30 个字段 → 考虑拆分。原则：一个实体一张表。

```
❌ 错误: 一张表 60 个字段，商品信息+订单+售后全堆一起
✅ 正确: 
   商品表 ──单向关联──→ 订单表
   订单表 ──单向关联──→ 售后表
   通过查找引用自动拉字段
```

### 关联配置模式

```
表A (子表)                    表B (主表)
┌──────────────────┐         ┌──────────────────┐
│ 单向关联(type 18) │────────→│ (被引用)          │
│ 查找引用(type 19) │←────────│ 自动拉取字段       │
└──────────────────┘         └──────────────────┘

示例: 追踪表选日期 → 自动拉指标表的当日GMV/退款率
```

### 查找引用配置

查找引用需要指定:
1. 关联字段: 应用内指向主表的关联字段名
2. 查找字段: 要拉取主表的哪个字段（如「GMV」）
3. 条件筛选: 可选，如只拉近7天数据

## 自动化规则（在飞书 UI 配置，非 API）

### 三大触发器

| 触发类型 | 场景示例 |
|---------|---------|
| 记录满足条件 | 状态变更为"待处理" → 通知负责人 |
| 到达记录时间 | 预计完成日期前1天 → 提醒 |
| 定时触发 | 每天早9点 → 推送仪表盘到群 |

### 推荐自动化模板

| 业务场景 | 触发 | 动作 |
|---------|------|------|
| 新任务通知 | 新增记录 & 优先级=🔴严重 | 发飞书消息给负责人 |
| 到期催办 | 预计完成日期前1天 | 发消息提醒 |

## 模板场景速查

建表时直接从以下模板选字段组合：

| 业务场景 | 推荐字段组合 | 关键类型 |
|---------|------------|---------|
| **设计素材库** | 图片(附件)+风格标签(多选)+来源平台(单选)+原始链接(URL)+采集日期+质量评分(Rating)+设计用途(多选)+备注(文本) | Attachment, MultiSelect, Url, Rating |
| **项目管理** | 任务名+负责人(人员)+截止日期+状态(单选)+进度(Progress)+附件 | User, Progress, DateTime |
| **客户CRM** | 客户名+客户类型(单选)+跟进时间+状态+成交金额(Currency) | Currency, SingleSelect |
| **内容日历** | 发布日期+平台(多选)+内容类型+状态+链接(URL) | MultiSelect, Url |
| **库存管理** | 商品名+数量+安全库存+状态(单选)+关联订单(关联) | Number, SingleLink |
| **问题追踪** | 问题标题+优先级(单选)+分析(文本)+解决动作+预期效果+难度(单选)+负责人+执行进度+状态+效果评估(单选) | SingleSelect, Progress |

### 规则：一张表≥2个视图

#### 视图1：「管理视图」— 给老板/决策者看
- **筛选**：近 7 天 / 本月至今
- **显示字段**：只展示核心指标（GMV、订单数、退款率、转化率、店铺评分），隐藏技术细节
- **排序**：按日期倒序
- **冻结**：首列（日期列）
- **类型**：Grid（表格）

#### 视图2：「运营全览」— 给运营/执行者看
- **筛选**：无（全量数据）
- **显示字段**：全部字段
- **排序**：按日期倒序
- **类型**：Grid（表格）

#### 视图3（追踪表专用）：「看板视图」— 按状态分组
- **类型**：Kanban
- **分组字段**：状态/优先级
- **卡片显示**：标题 + 负责人 + 截止日期

### 创建视图（API 调用示例）
```
feishu_create_bitable_view(app_token, table_id, "老板看板", "grid")
feishu_create_bitable_view(app_token, table_id, "看板视图", "kanban")
```

## 仪表盘（Dashboard）

### 现状
飞书 API 不支持直接创建空白仪表盘。只能：列出 + 复制已有仪表盘。

### 策略
1. 在飞书 UI 中手动创建一个「模板仪表盘」（包含：指标卡、趋势图、表格块）
2. 用 `feishu_copy_bitable_dashboard` 复制模板
3. 复制后手动调整数据源

### 仪表盘最小可行组件
| 组件 | 数据 | 位置 |
|------|------|------|
| 指标卡 | 今日GMV、订单数、退款率 | 顶部横排 |
| 折线图 | 近7天GMV趋势 | 中部左侧 |
| 柱状图 | 各渠道GMV占比 | 中部右侧 |
| 表格块 | 待处理问题TOP3 | 底部 |

## 创建表格的完整流程

### Phase 1: 设计字段（先在纸上/对话中列出来）
1. 列出所有需要记录的信息
2. 按决策树确定每个字段的 type + ui_type
3. 金额 → Currency，百分比 → Progress，评分 → Rating，状态 → 单选+颜色

### Phase 2: 创建表
```
feishu_create_bitable_table(app_token, "表名", fields=[...])
```
在 fields 数组里直接用正确的 type/ui_type，不要建完再改。

### Phase 3: 写入数据
```
feishu_add_bitable_records(app_token, table_id, records=[...])
```
注意 Progress 字段值在 0-1 范围。

### Phase 4: 创建视图
```
feishu_create_bitable_view(app_token, table_id, "管理视图", "grid")
feishu_create_bitable_view(app_token, table_id, "看板视图", "kanban")
```

### Phase 5: 手动设置（API 不支持，需在 UI 操作）
- 条件格式/颜色规则（退款率>15%红色，转化率<1%红色）
- 仪表盘组件
- 高级权限
- 应用模式页面

## 常见错误与避免

| ❌ 错误做法 | ✅ 正确做法 |
|------------|-----------|
| 所有字段用 type=1（文本） | 金额→Currency，比例→Progress，状态→SingleSelect，人→User |
| 百分比字段值存为 15.3 | Progress 字段值存为 0.153（0-1范围） |
| 一张表 80 个字段 | 拆成多张实体表，用关联字段(type 18)连接 |
| 只建一个默认视图 | 至少建管理视图+运营视图+看板视图 |
| 状态字段用文本手写 | 用 SingleSelect+颜色，只能选不能打错字 |
| 更新单选字段时只传新选项 | 必须传完整 options 列表（会覆盖） |
| 公式字段嵌套 10 层 IF | 拆成辅助字段分步计算 |
| 上手就建 20 张表 | MVP 先行：核心表+核心字段，跑通再扩展 |
| 公式用 {字段名} | 用 [字段名]，中括号，精确匹配含符号 |
| 负责人用 Text 存名字 | 用 User (type 11)，支持 @mention 和自动通知 |
| 跨表复制粘贴数据 | 用关联(type 18)+查找引用(type 19)自动拉 |
| 素材图片存本地不发飞书 | 用 Attachment(type 17)存在表格里，统一浏览 |
| 手动发通知催进度 | 用自动化规则，状态变更自动通知 |
| 每次手抄指标值 | 用公式(type 20)自动计算，只读防误改 |

## 设计检查清单

建任何飞书表之前过一遍：

- [ ] 所有金额字段用了 Currency？
- [ ] 所有比例/比率字段用了 Number（后续改 Progress）？
- [ ] 所有状态/分类字段用了 SingleSelect + 颜色？
- [ ] 负责人能改成 User (type 11) 吗？（需要飞书 open_id）
- [ ] 有需要自动计算的指标 → 加了 Formula 字段？
- [ ] 图片/文件需要存 → 加了 Attachment 字段？
- [ ] 有跨表引用需求 → 加了关联+查找引用？
- [ ] 最少 2 个视图？（管理视图 + 运营视图）
- [ ] 关键字段设了必填约束？
- [ ] 有重复操作需要自动化规则？
